Concepts

Automatiko is based on concept of modeling business logic and decisions to capture and express business requirements. It is all about understanding and explaining (as model) what the service is about.

Process

Process (aka workflow) is the main entry point of the service. It is (depending on the language) a flow chart (BPMN2) or a text based (json/yaml for serverless workflows).

Process consists of activities that are executed within the life time of every process instance. These activities have different meaning depending of their type or place in the model. To just name few

  • activities that invoke services ('service task' in BPMN or 'operation state' in serverless workflow)

  • events that either capture events or produce events

  • subprocesses that creates new process instance of another process to build hierarchy of process instances and take advantage of process composition pattern ('call activity or reusable subprocess' in BPMN or 'subflow' in serverless workflow)

Identification and description

Process has to have assigned id. Id serves as unique identification of your process and as such should be

  • short (usually single word in case of multiple words they should be written in camel case)

  • descriptive (carry most important information what the process is dealing with)

The reason why these rules are describe is that process id becomes your service identification as well. With that said, it will be the top level resource of the REST api for the process.

Best practice is to thing about process id in context of the domain it is dealing with. For example when process is managing orders then process id should be orders

In addition to id, process have name that should be provided as more lengthy description of the logic encapsulated in the process definition. It should not be to long so a sentence or two, for more documentation like needs process allows to have add extra information on various levels.

Documentation can be added to:

  • process itself

  • every activity of the process

  • every data object (aka variable) of the process

  • every event defined in the process

Using documentation is recommended as it enhances service description automatically which allows service consumers to better understand intention of the service and by that ease the integration.

Documentation within the process is not supported in serverless workflows at the moment.

Versioning

Versioning of a process should be considered from the very beginning as it might have severe impact on consumers of the service and maintainability of the service.

Following are basic principles to version handling in Automatiko

  • version can be assigned to any process (public and private)

  • version will be part of the service contract - in REST api it will prefix the paths e.g. /v1/orders or /v1_2/orders for versions 1 and 1.2 respectively.

  • multiple versions of the process should be kept in the same service, meaning new version should be a copy of the previous one

Change version only when there is good reason for it. As a rule of thumb it is recommended to not change version of the process if changes are backward compatible. In other words, change version only when the modifications to the process make the existing instances imposible to continue.

Type of processes

Process types are currently only supported in BPMN language and not in serverless workflow.

Type of process defines visibility of given process on the service API. There are two types of processes

  • Public - these are exposed as top level service endpoints

  • Private - these are hidden from the service API meaning instances of such processes cannot be started from outside.

Private processes are perfect fit for subprocesses to manage creation of new instances to be limited to the so called owning process instance.

Process data

Process data is the most important part. This defines what is the actual domain of the service that is being modeled. In the end all services are meant to deal with data and as such alter it.

Properly defining data model of the process is extremely important to the service and the contract it will expose. To help with that Automatiko promotes use of so called tags to annotate and enhance data handling.

Data object tags

Process data also known as data objects or variables can be annotated with tags to enable additional features

Tag name Description

input

Marks given data object as input that should be given upon starting new instance

output

Marks given data object as output that will be returned always when instance is altered or completed

internal

Marks given data object as internal that should not be visible from outside

notnull

Marks given data object as non nullable, meaning once it was assigned a value it cannot be reset to null

readonly

Marks given data object as read only meaning that it cannot be changed after being set once

required

Marks given data object as required to start new instance

auto-initialized

Marks given data object to be auto initialized upon start meaning it guarantees that the value will be set

business-key

Marks given data object to be used as business key of the instance, applies only for the start of the new instance

initiator

Marks given data object to be the source of initiator of the instance instead of relying on security context
Serverless workflow does not define data objects explicitly and allows to have data pushed on demand and thus do not make use of tags.

Process instance

Process instance is an individual instance of a given process. This is a runtime representation of just one item being processed according to the given definition.

Depending on the process definition (how it was modeled) it can be a short lived or it can span hours, days, months and more.

Each process instance is uniquely identified with generate id (of a form of UUID). This id remains unchanged during entire life time of the instance.

Business key

As an alternative id of process instance, a business key can be assigned. Business key when assigned will remain unchanged (same as id) during entire life time of the instance but can be used exactly the same as id. Meaning it can be used to reference given instance from outside with service API.

Even though business key is assigned the generated id can still be used to reference the instance.
Business key must be unique within the scope of a process (or version of the process). That means you can have only one active instance with given business key.

Tags

Similar to tags on process data, process itself can define tags. Tags allow to put extra information (besides business key) that are used for correlation purpose.

Tags can be both simple (constant) values that won’t change over time or expressions that are evaluated every time process instance change.

Additional feature of process tags (compared to process data tags) is that tags can be added and removed during entire life time of process instance.

Tags defined in process (definition) cannot be removed, only tags added on top of active process instance can be removed.

Decision

Decision is defined to capture decision logic within the process. It is not exposed as separate service but instead is invoked from within a process. Decision in Automatiko are defined as DMN diagrams which goes hand in have with BPMN to have both defined in graphical way.

Decision requires following to be properly integrated

  • decisions are identified by model and namespace and optionally decision or decision service

  • decisions must define inputs which will be then mapped from the process data

  • decisions define various decision logic constructs such as

    • literal expression to encapsulate expression like decisions

    • decision tables

Decision results are then mapped (by name) to process data.
