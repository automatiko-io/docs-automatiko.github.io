Use Cases

Event streams

Event streams are very popular to represent changes to data. They allow individual modifications to be represented as atomic operations that are

  • associated with timestamp

  • uniquely identifiable

  • carry all relevant information to the objects they reference

Workflows are perfect fit into processing event streams as they can provide additional view of the domain changes equipped with business logic to allow description why and how the data has changed.

Above diagram illustrates simple order processing use case that is backed by event stream. In this particular example Apache Kafka is used as event log broker but any other solution could replace it. Automatiko seamlessly integrates with Apache Kafka and thus it is used in the examples that cover supported use cases.

Events in this use case are represented as Key-Value Records. Where key is the unique identifier e.g. order number and values are appended for each change to particular order. From workflow point of view, new instance of the workflow is started as soon as first event is published with given order number. The order number is considered as business key - an unique identifier of the workflow instance.

Automatiko allows to use business key as alternative id of the workflow instance and by that can be used to reference given instance via API.

Once the instance is started, every other event published to the event stream with the same key will correlate to already active instance and push event data to be able to progress according to business logic defined for given event.

What is also important is that business key is not the only correlation that Automatiko supports. Workflow tags are also used as correlation. This can enable additional features such as receiving events that are associated with customer who placed the order, e.g. when customer has updated he shipping address all ongoing orders should be notified about this change to make sure they will be sent to the correct location.

To see this use case implemented with Automatiko have a look at Order example

Internet of Things

IoT use case is one of the underestimated when it comes to workflows and at the same time is one of the most natural fit for workflows. Data produced by sensors do not have too much value on its own. But associated with business logic that can produce very powerful solution.

Automatiko allows to cover this use case in number of ways

  • minimal - that allows to build automation directly on IoT devices such as mini computers (Raspberry PI) or IoT gateways.

  • private network deployments - automation for IoT devices remains in the same network as the devices and sensors itself for maximum security and cost efficiency

  • cloud deployments - automation is placed in the cloud that then enables easier access to it from any place and any device.

  • mixed of all the above

The backbone for IoT use case with Automatiko is the MQTT broker that acts as hub that allows different components to exchange information in the most efficient way.

Above diagram presents a simple case for collecting sensor data in kind of home automation solution. Different types of sensors publish their data into dedicated MQTT topics

  • home/roomxxx/temperature

  • home/roomxxx/humidity

  • home/roomxxx/window

where roomxxx is the identifier of a room of the house such as living room, bedroom, bathroom, etc.

Automatiko allows to define a workflow that can collect sensor data and associate them with the room it was published for. In this case the room identifier becomes the correlation so each room publishing its data will be kept in single workflow instance that allows to build logic around data buckets. In addition to that Automatiko comes with notion of time windows that the data will be collected for. This essentially means that given workflow instance will be active no longer than the time window defines. So one could define business logic to collect sensor data for an hour and after one hour such workflow instance will be discarded. Subsequent sensor data for the same room will create new workflow instance and new time window.

To see this use case in action, have a look at the Water leaks detection example

Human centric workflows

Human centric workflows is one of the most common use cases for workflows. It was often promoted by traditional Business Process Management (BPM) solutions. And indeed, it is still a valid use case as in literally every business there is human interaction of some kind. Vacation approval, loan application, insurance application are examples of human centric workflows.

It essentially focuses on automation of human required actions. In some cases it is to make sure the most appropriate individuals are found to perform work in other it’s more important that there is a timely response to the assigned work. All depends on the domain given workflow is put in.

Above use case handles support cases - incidents reported by end users. There can be many channels that such incident can be reported

  • via email

  • via web form

  • via phone call

  • other

Regardless of how the data is provided the handling of the case will be similar or even exactly the same. The case needs to investigated, classified, assigned to support engineer and finally the case needs to be resolved. As in the end this is the business goal of such workflow.

Automatiko provides support for handling all these stages and efficiently manages user communication such as

  • assignment of users via individual or groups assignments

  • notification over email or other channels to make sure assigned actors are aware of awaiting tasks.

In addition to that having clear visibility into current state of cases is critical. Having option to build dashboards like entries can significantly improve visibility and awareness for business stakeholders. In the end it is all about improving how human actors work with their daily routines and automate whatever is possible.

To see this use case in action, have a look at the Support case example

Service orchestration

Service orchestration is another traditional use case coming from SOA (Service Oriented Architecture) times. Although it’s somehow dated it is certainly relevant nowadays too. It might be even more relevant than before and that is because of the raise of microservices and event driven architectures.

These architectures put in front the communication between services, despite what communication protocol is used (REST/HTTP, messaging, other) there is always a need to coordinate this communication.

And for exactly that reason workflows come quite handy to be able to define the interactions between services in declarative way. In addition to that it comes with built in features to ensure such communication is done easily and efficiently. To name just few of these features

  • easy integration with various communication protocols

    • REST backed by OpenAPI

    • messaging with out of the box support for Kafka, MQTT

    • integrates with various systems via components backed by Apache Camel

  • declarative security

  • error handling

  • retry for failed invocations

Above example shows a simple service orchestration aka coordination of services that build up rescue personnel information service based on weather forecast.

It bundles various features of workflows to provide fully featured coordination between independent services.

  • instances of the workflow are automatically created based on time intervals - every hour

  • various services are invoked during execution to fetch data (weather service) or delegate (alert service, information service, notification service)

  • each and every service can be either not accessible or returns unexpected response so workflow is equipped with error handling and retry mechanism to improve fault tolerance

Service orchestration is not just about calling services or consuming/producing events, it is also about control logic. What to do based on various condition is one of the most powerful assets o workflows in this use case. One example of such control logic is how to handle cases that require undoing already performed operations. Workflows come with (again) built in feature called compensations. Compensation allows to define additional logic that should be invoked when decision about compensating is taken. This is all defined within the workflow model and by that made explicit to all involved parties.

To see this use case in action, have a look at the Rescue information example

Data base record processing

As we know most of the data, nowadays systems deal with are stored in data bases. Regardless of their type, this data is always a subject of modification. Systems have to create, read, update and delete data stored in database. In most of the case all this logic is custom but has well defined life cycle. This life cycle is defined by business context it resides in. All this is actually a defined process on how data (and by whom) can be modified.

So for that reason, workflows make a good fit for it as well. It is not to replace custom code but is to enhance or simplify it.

Above example to handle vacation requests submitted by employees illustrates how data base records can be used from within workflow to manage its lifecycle.

The use case itself is more about how data is being managed from storage point of view but still being part of workflow. Every type of data (here person record and vacation request record) is stored securely in database. But most the modifications to it are controlled by workflow. Here the benefits of workflow like isolation and security are clearly visible - each instance is responsible for given set of data base records.

On top of this, data are kept in db as any other records (that are not managed by workflow) so they are accessible with any other tool available. It can be used to build reports, dashboards, backups etc.

To see this use case in action, have a look at the Vacation requests example
