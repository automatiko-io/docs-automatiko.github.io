What’s Automatiko?

Overview

Automatiko is a toolkit that takes advantage of well established and known languages to build (but not limited to) self contained services. As the name might suggest its main goal is to enable users to automate their daily tasks, be it processing of emails, perform schedules actions, collect and transform data and many many more.

By "well established and known languages" Automatiko means

  • BPMN2 (business process modeling and notation)

  • DMN (decision modeling and notation)

In addition to these already known in the market, a relatively new player is also supported in Automatiko

  • Serverless Workflow ( a Cloud Native Computing Foundation project)

Automatiko promotes above mentioned languages as programming languages that can be used to build production ready services. Takes advantage of various constructs of these languages to provide a service that is:

  • self described

  • API contract enabled

  • secured

  • easy to managed

  • integrated with other services

Why should one use Automatiko?

Before answering this fundamental question, where is one even more important to start with

Why should one use workflow?

Workflows in general are meant to capture business logic in structured way. That is either graphical (like BPMN and DMN do in a form of flow charts) or declarative as it is done in serverless workflow. This gives us first reason to use workflows

Visibility

Visibility of the business logic that can be applied in both design (development time) to make sure the business logic is expressed the way business demands it, at runtime to illustrate current state of individual (or aggregated) instances.

Another important aspect of workflows is that they define data objects that are required for execution. That applies to workflow as a whole but also to individual activities such as service tasks or operations, user interactions. This can be seen as second reason

Domain data contract

Domain data contract allows to clearly define what is an input and what is an output of a given activity or the workflow. With this at hand your domain business logic is equipped with one of the most powerful tools - self description.

Next item on the list is the isolation and security of your domain data. As workflows define the domain data model (as stated above) at the same time, workflows are responsible for security around accessing these data. Not just from authorization point of view but also (what is in many case equally important) isolation.

Isolation

Isolation is ensured on individual levels without any extra effort. Workflows are built around definition and instances. Definition is a single artefact that is then initialized for each individual set of data which then forms an instance. Each instance is kept isolated from others while still providing access to each at any point in time.

Yet another argument to use workflows is to use the tools that are available and proven instead of attempting to build it from scratch (which is certainly possible but in many cases not worth the effort). Such tools that workflow come out of the box with are:

  • service invocation (orchestrating services via REST)

  • message exchange (Apache Kafka, MQTT, JMS and many more)

  • user interaction (assigned task to groups and individual, notification about pending work)

  • distributed time tracking (when workflow need to work for a given amount of time)

  • data driven actions (reacting to data changes to trigger actions)

  • and many many more

Workflow toolbox

Out of the box tools that come with workflows that are directly available for use regardless of what domain they are used in.

So this gives just few arguments to consider why use of workflows makes sense. As usual, it all depends on particular use case but as in life everything is a process so why not make use of the real life approach to it and model business logic as they are…​

So why to use Automatiko?

Automatiko was designed and built with above principles in mind to allow developers an easy and yet powerful way of defining their business logic. It was tailored to provide excellent support for number of use cases

  • event streams

  • Internet of Things

  • human centric workflows

  • service orchestration

  • data base record processing

All these use cases you can examine in the Use Cases chapter and see them in action.

Automatiko is completely open source under Apache licence. Automatiko is written in java and takes advantage of all the modern frameworks and libraries that allow it to run on pretty much any environment to name few

  • local environments on any of the popular OS - Windows, Linux, Mac

  • IoT devices such as Raspberry PI

  • Cloud deployments

    • Kubernetes

    • K3S (and K3D)

    • OpenShift

    • AWS

It’s main aim is to make BPMN (and serverless workflow) and DMN as programming languages that provide value to build services that automate your business logic, regardless what the domain is.

It’s not intended to replace other programming languages but more of embrace out of the box features of workflows that provide significant value with little effort.

Give it a try and share your feedback

So enough of talking, take it for a spin with examples ready for use without much of a hassle. Just head to Getting started chapter and try it! Don’t forget to share your feedback as this will allow us to make it even better.
